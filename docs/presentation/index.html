Why Confidential Containers?

Welcome! Are you curious how a pod can keep secrets from the very cluster that runs it? What lives inside a confidential container? Or why an attestation service would rather return a 403 than take your word for anything?

This repo stands up OpenShift Confidential Containers (CoCo) on AMD SEV‑SNP bare metal. Every capability is first proven on a disposable single-node test rig — spun up, measured, torn down — and only then applied, unchanged, to the real air-gapped multi-node cluster. No peer-pods, no cloud shims: the worker's own RHCOS kernel is the SNP host.

The main question we want to answer: why does a secret release only when the hardware proves what's running — on a cluster with no internet at all?

Let's take a closer look.

05 · The ladder

Prove It, One Rung at a Time

Each capability is a rung: proven on the rig, then applied to production. A rung is "done" only when it reproduces from written steps on a fresh node and its negative test — the denial — passes and auto-reverts. make negative-test WHICH=all

Proven

Rung A · Secret Release

rung-kbs

Attestation gates a credential. Air-gapped attest via the VCEK OfflineStore → the KBS releases the secret. No attestation → withheld with a 403.

Proven

Rung B · Measurement

rung-rvps

A populated snp_launch_measurement gates release. Tamper with the initdata → 403; the untampered control still releases. Proven 2026‑07‑01.

Air Gap · OfflineStore

kbsLocalCertCacheSpec

Proof the cache is load-bearing: swap the VCEK for a wrong cert and attestation fails with a 401 — not a silent fallback to AMD's public KDS.

Rung C · Signed Image

image_security_policy rejects unsigned or tampered images. Blocked on the rig only because the minimal mirror registry doesn't serve the Quay signature extension.

Rung D · Encrypted Image

Wrong measurement → the decryption key is withheld and the pod never starts. Manual for now: host encrypted-layer pre-pull is blocked upstream (cri-o #10084).

About this breakdown

What Is This Repo?

What is this repository?

A complete, reproducible path to OpenShift Confidential Containers on AMD SEV‑SNP bare metal: Terraform for the metal, Ansible for the bastion and install, a Kustomize GitOps tree shared between rig and production, and a Makefile that drives the proof gates. A fully manual, provider-neutral guide lives in docs/install-guide.md.

Why a disposable rig?

Confidential computing fails in expensive, hardware-shaped ways. The rig — one hourly SNO node plus a second Trustee cluster, egress-firewalled to simulate the air gap — lets every capability be proven and torn down before it ever touches customer metal.

What changes between rig and production?

Only the hardware-bound parts: VCEK certificates, RVPS reference values, TLS identity, and the exact initdata bytes measured by the TEE. The manifests, operator ordering, policy shape, and workload patterns promote unchanged through GitOps overlays.

Where do I start?

make help lists the targets. The very first gate is make verify-snp-host NODE=<node> — rung‑0 proves the silicon before a single manifest is applied. Then follow the runbooks, rung by rung.
