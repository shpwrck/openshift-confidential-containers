Why Confidential Containers?

Welcome! Are you curious how a pod can keep secrets from the very cluster that runs it? What lives inside a confidential container? Or why an attestation service would rather return a 403 than take your word for anything?

This repo stands up OpenShift Confidential Containers (CoCo) on AMD SEV‑SNP bare metal. Every capability is first proven on a disposable single-node test rig — one hourly Latitude.sh machine, spun up, measured, torn down — and only then applied, unchanged, to the real air-gapped multi-node cluster. No peer-pods, no cloud shims: the worker's own RHCOS kernel is the SNP host.

The main question we want to answer: why does a secret release only when the hardware proves what's running — on a cluster with no internet at all?

Let's take a closer look.

04 · First, prove the silicon

Rung-0: The Gate Before Everything

Confidential computing fails in expensive, hardware-shaped ways — so before a single manifest is applied, one script must go green. make verify-snp-host proves the machine can actually do SEV‑SNP as a host.

$ make verify-snp-host NODE=rig-01
AMD CBS SEV-SNP SUPPORT ON SMEE ON SEV-ES ASID LIMIT > 0 RMP TABLE ON MEMORY INTERLEAVING OFF

Kernel can't? → image problem

If the kernel lacks SNP-host support (needs ≥ 6.11, CONFIG_KVM_AMD_SEV=y), that's a boot-image problem. Boot a newer kernel — it is not a provider veto.

Kernel can, but sev_snp=N? → BIOS

The toggle is off. Open the IPMI console into AMD CBS and apply the recipe above — PSP Error: 0x3 means Memory Interleaving is still enabled.

Both fine, still dead? → veto

Kernel-capable, BIOS on, and SNP still won't init: old PSP firmware or a genuine provider veto. Only this case justifies abandoning the bare-metal path.

05 · The build

Fill the Bastion, Then Seal the Gap

An air-gapped cluster can't fetch anything later — so everything it will ever need is staged on a bastion host while it's still connected. Then the line to the internet is cut, and the install runs entirely from the private side.

07 · Negative tests

Now Let's Break It — On Purpose

A denial you've never seen is a denial you can't trust. Every rung ships with a scripted negative test: break exactly one thing, demand the refusal, then auto-revert the rig to baseline. make negative-test WHICH=all

WHICH=What we breakWhat must happen
rung-kbs Ask for the secret with no valid attestation — a bare request. The attested control still releases. Withheld · 403 — the KBS gates release on attestation, not on network position.
rung-rvps Tamper the measured initdata under a restrictive measured-initdata policy (applied, then reverted). Untampered control releases. Withheld · 403 — valid attestation, wrong measurement. Proven on the rig 2026‑07‑01.
air-gap Swap the VCEK in the OfflineStore for a wrong certificate. Attestation fails · 401 — not a silent fallback to AMD's public KDS. The cache is load-bearing.
rung-signed Launch an unsigned / tampered image under image_security_policy. Image rejected — pod never admitted.
rung-encrypted Present a wrong measurement when requesting the image decryption key (manual rung). Key withheld — the layer never decrypts; the pod never starts.

Each denial backs up what it swaps and auto-reverts · make repro-loop runs deploy → positive → negative for every rung, resumably

08 · One tree, two worlds

The Same GitOps, Rig and Production

The whole point of the rig: what gets proven is what gets promoted. One Kustomize base fans out into rig overlays and customer overlays — only the hardware-bound parts change.

Portable — promoted unchanged

  • Manifests and operator ordering
  • Policy shape (Rego + Gatekeeper)
  • KBS resource definitions
  • Workload patterns and runtime classes

Hardware-bound — regenerated on customer metal

  • VCEK certificates (per-socket)
  • RVPS reference values (Veritas, on-target)
  • TLS identity + Trustee URL
  • The exact initdata bytes the TEE measures

Prove It, One Rung at a Time

Each capability is a rung: proven on the rig, then applied to production. A rung is "done" only when it reproduces from written steps on a fresh node and its negative test passes. Status below is from the rig — disconnected SNO on EPYC Genoa, last proven 2026‑07‑01.

rung-rvps

A populated snp_launch_measurement gates release — init_data == sha256(initdata bytes) confirmed as the measured HOST_DATA. Tampered initdata → 403. Proven 2026‑07‑01.

Air Gap · OfflineStore

kbsLocalCertCacheSpec

Proof the cache is load-bearing: swap the VCEK for a wrong cert and attestation fails with a 401 — not a silent fallback to AMD's public KDS.

Rung C · Signed Image

rung-signed

Upstream-blocked

Rung D · Encrypted Image

rung-encrypted

Wrong measurement → the decryption key is withheld and the pod never starts. Manual for now: host encrypted-layer pre-pull is blocked upstream (cri-o #10084) — a skipped D is not a failure.

About this breakdown

Questions People Actually Ask

Why a disposable rig?

Confidential computing fails in expensive, hardware-shaped ways. The rig — one hourly SNO node plus a second Trustee cluster, egress-firewalled to simulate the air gap — lets every capability be proven and torn down before it ever touches customer metal.

What counts as "proven"?

Two things, both mandatory: the rung reproduces from written steps on a fresh node, and its negative test — the denial — passes. A demo that works once is not a proof; a refusal you've never seen is not a control.

What changes between rig and production?

Only the hardware-bound parts: VCEK certificates, RVPS reference values, TLS identity, and the exact initdata bytes measured by the TEE. The manifests, operator ordering, policy shape, and workload patterns promote unchanged through GitOps overlays.

Can I drive it from macOS?

Yes — the operator scripts run on stock macOS (bash 3.2 + BSD userland, no GNU coreutils needed). A runbook covers which scripts run on your workstation and which belong on the bastion.

What about Intel TDX?

SEV‑SNP first. TDX arrives later as an additive overlay on the same GitOps shape — with one caveat: its air-gapped attestation story (the equivalent of the VCEK OfflineStore) has to be proven the same way, negatives and all.

Where do I start?

make help lists every target. The very first gate is make verify-snp-host NODE=<node> — rung‑0 proves the silicon before a single manifest is applied. Then follow the runbooks, rung by rung.
